需求工程與系統塑模

以「線上點餐外送系統」為範例，帶你掌握軟體工程核心概念。
涵蓋 Chapter 4（需求工程）與 Chapter 5（系統塑模）完整內容。

📚 課程說明

本教學網站以一個貫穿全課程的範例系統，幫助你理解抽象的軟體工程概念。每個章節都會用同一個系統來示範，讓你看到需求如何一步步被定義、分析、塑模成具體的設計。

🍔 範例系統：線上點餐外送平台

想像一個類似 Uber Eats 或 foodpanda 的系統，具備以下情境：

顧客端

瀏覽餐廳、搜尋餐點、加入購物車、下單、線上付款、追蹤外送進度、評價餐廳

餐廳端

管理菜單、接收訂單、更新製作狀態、查看營收報表、設定營業時間

外送員端

接受派單、導航取餐、確認送達、查看收入紀錄

管理員端

審核餐廳上架、處理客訴、管理使用者帳號、查看系統數據

為什麼選這個系統？ 每位同學都用過外送平台，容易理解業務場景。而且這個系統有豐富的角色互動、清晰的狀態轉換、多樣的類別關係，非常適合用來學習所有 UML 圖。

Chapter 4

4-1 功能需求 vs 非功能需求

📚 概念說明

系統的需求（Requirements）描述系統應提供的服務及其運作限制。需求通常分成兩大類：

  • 功能需求（Functional Requirements）：系統「應該做什麼」，描述系統提供的服務、對特定輸入的反應、以及在特定情況下的行為。
  • 非功能需求（Non-functional Requirements）：系統提供服務的「限制條件」，通常與效能、安全性、可用性、可維護性等整體品質相關。

非功能需求通常比個別功能需求更重要 — 如果非功能需求不符合，可能讓整個系統無法使用。

🍔 點餐系統範例

功能需求範例

編號需求描述角色
FR-01顧客必須能依餐廳名稱或菜色類別搜尋餐廳顧客
FR-02系統必須在顧客下單後自動通知餐廳系統
FR-03餐廳必須能更新菜單品項與價格餐廳
FR-04外送員必須能在 App 上查看取餐地址與送餐地址外送員
FR-05系統必須在訂單完成後讓顧客對餐廳評分（1-5 星）顧客

非功能需求範例

類別需求描述可測量指標
效能系統在午餐尖峰時段必須承受高負載每秒處理 ≥ 1,000 筆並發請求
可靠性系統必須保持高可用性年可用率 ≥ 99.9%（停機 ≤ 8.76 小時/年）
安全性付款資訊必須加密傳輸與儲存符合 PCI DSS 標準
回應時間搜尋結果必須快速回應搜尋回應時間 ≤ 2 秒
可用性新使用者必須能快速上手首次使用者 5 分鐘內完成下單

非功能需求的三大來源

產品需求

來自產品本身的品質要求，例如：回應時間 ≤ 2 秒、App 大小 ≤ 100 MB

組織需求

來自公司政策與流程，例如：必須使用 Git 版本控制、每週發布一次更新

外部需求

來自法規或外部標準，例如：遵守個資法、符合食安法規

📜 重點整理

  • 功能需求描述系統「做什麼」，非功能需求描述系統「如何做」以及限制
  • 非功能需求應該是可量測的，避免模糊的目標敘述（如「容易使用」）
  • 非功能需求來自三大來源：產品、組織、外部
  • 需求規格應具備完整性（涵蓋所有需要）與一致性（不互相矛盾）
  • 使用者需求（高層描述）與系統需求（詳細規格）是不同層次

✎ 練習題

這是功能需求。它描述的是系統應該提供的一項具體服務（即時追蹤功能），而不是對服務品質的限制。

如果改成「即時位置更新的延遲不得超過 3 秒」，那就是非功能需求（效能限制）。

改寫範例：

  • 「餐廳列表頁面的載入時間不得超過 1.5 秒」
  • 「搜尋功能在 10,000 筆資料中的查詢回應時間 ≤ 500 毫秒」
  • 「訂單提交後，系統確認回應時間 ≤ 2 秒（在 95% 的情況下）」

關鍵在於用具體數字取代模糊形容詞，並指定測量條件。

屬於外部需求。這是來自系統外部的法律法規所衍生的需求，開發團隊必須確保系統合法運作。

Chapter 4

4-2 需求抽取技術

📚 概念說明

需求抽取（Requirements Elicitation）是了解關係人的工作內容、以及他們如何使用新系統的程序。這個過程並不容易，因為：

  • 關係人經常不知道可以要求系統做什麼
  • 不同關係人使用各自的術語與行話
  • 不同關係人有不同（甚至衝突的）需求
  • 政治因素與環境變動會影響需求

需求抽取程序包含四個活動：發現需求 → 分類與組織 → 排列優先順序與協調 → 製作文件，不斷循環。

🍔 點餐系統範例

技術一：訪談

最常用的需求抽取方法，可分為限定訪談（事先定義好問題）與開放式訪談（自由討論）。

情境：訪談一位餐廳老闆

限定訪談問題：

  • 您目前接單到出餐的平均時間是多久？
  • 您希望系統如何通知您有新訂單？
  • 您需要什麼樣的營收報表？

開放式訪談發現：

  • 老闆提到「尖峰時段常常漏單」 → 發現系統需要訂單提醒與逾時警告功能
  • 老闆說「有些菜用完就沒了」 → 發現需要即時庫存管理功能（原本沒想到）

訪談的限制：領域專家可能覺得某些知識是「常識」而不會主動提起。例如餐廳老闆不會特別說「出餐要照順序」，但這對系統設計很重要。

技術二：民族誌法

民族誌法（Ethnography）是觀察法 — 分析人員融入實際工作環境，觀察日常運作。

情境：分析人員在餐廳廚房觀察一整天

觀察發現：

  • 廚師看到訂單後會先分類（冷飲先做、熱食最後），不是照下單順序 → 系統需支援訂單分類排序
  • 外送員到了但餐還沒好，會佔用出餐區 → 需要餐點準備完成通知外送員功能
  • 尖峰時段老闆會暫時關閉某些品項 → 需要暫停接單/暫停品項功能

技術三：故事與情境

透過真實故事描述使用情境，讓所有人都能理解系統的使用方式。

情境：顧客下午茶訂餐

初始狀態小明打開 App，已登入帳號，所在位置為學校宿舍
正常流程1. 系統依據 GPS 定位顯示附近餐廳
2. 小明選擇「好喝茶飲店」
3. 小明選擇珍珠奶茶（大杯、微糖、少冰）加入購物車
4. 小明確認訂單金額，選擇 Line Pay 付款
5. 系統建立訂單並通知餐廳
6. 小明在 App 追蹤訂單狀態
可能問題餐廳無法接單（已打烊）→ 系統提示「該餐廳目前未營業」
付款失敗 → 系統提示重試或更換付款方式
其他活動餐廳收到訂單後開始準備、系統同時尋找附近外送員
完成狀態外送員送達，小明確認收餐，系統請小明評價

📜 重點整理

  • 訪談是最基本的技術，結合限定與開放式效果最好
  • 民族誌法能發現訪談中遺漏的隱性需求，但耗時且不適合發現新功能
  • 故事與情境用具體例子描述使用流程，便於所有人理解
  • 情境應包含：初始狀態、正常流程、可能問題、其他活動、完成狀態
  • 需求抽取是反覆循環的過程，不是一次完成

✎ 練習題

建議結合訪談和民族誌法：

  • 訪談外送員了解他們的痛點（如：找不到地址、等餐太久）
  • 實際跟隨外送員工作一天（民族誌法），觀察他們在不同情境下的行為，例如下雨天、樓層不明確、電話聯繫不上顧客等

民族誌法特別重要，因為外送員的工作環境複雜（在路上、不同餐廳、不同社區），訪談可能無法完整涵蓋。

初始狀態顧客已收到餐點，發現品項與訂單不符
正常流程1. 顧客在 App 點選「訂單問題」
2. 選擇「餐點錯誤」並拍照上傳
3. 系統自動通知餐廳
4. 餐廳確認問題後選擇「同意退款」
5. 系統將款項退回顧客帳戶
可能問題餐廳不同意退款 → 轉交平台客服處理
其他活動系統記錄此筆客訴供後續分析
完成狀態退款完成，顧客收到退款通知

Chapter 4

4-3 需求規格制訂

📚 概念說明

需求規格制訂（Requirements Specification）是將使用者需求與系統需求撰寫成正式文件的過程。常用的撰寫方式有：

  • 自然語言：表達容易、通用，但容易模糊不清
  • 結構化自然語言：使用標準範本，減少模糊性
  • 表格與圖形：適合描述多種情況的對應處理

需求規格的不精確會導致客戶與開發團隊的爭執。模棱兩可的需求，開發人員會朝容易實作的方向解讀，但這可能不是客戶想要的。

🍔 點餐系統範例

自然語言需求（模糊版）

「系統應該讓使用者能搜尋餐廳。」

這個需求有什麼問題？搜尋的方式是什麼？可以搜名稱嗎？搜菜色呢？搜尋範圍呢？這些都沒有定義。

結構化需求（清晰版）

需求編號FR-01
功能描述餐廳搜尋功能
輸入搜尋關鍵字（餐廳名稱、菜色名稱、菜系類別）、使用者 GPS 位置、搜尋半徑（預設 3 公里）
輸入來源顧客 App 介面
輸出符合條件的餐廳列表（含名稱、評分、距離、預估送達時間）
輸出去向顧客 App 搜尋結果頁面
前置條件顧客已登入且已授權定位
後置條件搜尋結果依距離排序，每頁顯示 20 筆
副作用系統記錄搜尋紀錄供推薦演算法使用

表格式需求：外送費計算

當情況有多種條件時，表格特別有用：

訂單金額距離時段外送費
≥ $300≤ 3 km一般免運費
≥ $300≤ 3 km尖峰$15
≥ $300> 3 km一般$10/km
< $300≤ 3 km一般$30
< $300> 3 km一般$30 + $10/km
任何任何尖峰上述費用 + $15

需求確認

寫完需求規格後，必須透過以下方式確認：

  • 有效性檢查：需求是否反映真正需要？
  • 一致性檢查：需求之間是否矛盾？
  • 完整性檢查：是否遺漏重要功能？
  • 實現性檢查：以現有技術是否能實作？
  • 可驗證性：需求寫法是否能被測試？

📜 重點整理

  • 自然語言易懂但模糊，結構化範本能大幅降低歧義
  • 結構化需求應包含：描述、輸入/輸出、前置/後置條件、副作用
  • 多條件對應時使用表格最清楚
  • 需求確認有五種檢查：有效性、一致性、完整性、實現性、可驗證性
  • 需求會持續變更，要有需求變更管理程序（問題分析 → 影響評估 → 實作變更）

✎ 練習題

問題是太模糊。「多種」是哪些？應改為結構化描述：

「系統應支援以下付款方式：(1) 信用卡（Visa/MasterCard/JCB）、(2) Line Pay、(3) 貨到付款（現金）。每種付款方式的交易確認時間不得超過 5 秒。」

需求編號FR-10
功能描述顧客取消訂單功能
輸入訂單編號、取消原因（選填）
輸入來源顧客 App 訂單詳情頁面
輸出取消確認通知、退款金額資訊
輸出去向顧客 App + 餐廳端通知
前置條件訂單狀態為「已下單」或「已接單」（餐廳尚未開始製作）
後置條件訂單狀態更新為「已取消」，款項依退款政策退回
副作用若已接單則通知餐廳停止製作

Chapter 4

4-4 Use Case 初探

📚 概念說明

使用案例（Use Case）是以圖形模型和結構化文字，描述使用者與系統互動的方式。它是 UML 的基本功能之一。

  • 行動者（Actor）：與系統互動的人或外部系統，用人形符號表示
  • 使用案例：系統提供的一項功能，用橢圓表示
  • 每個使用案例代表一個獨立的互動任務

Use Case 圖只是概述，通常還需要搭配文字描述或序列圖來補充細節。這部分會在 Chapter 5 詳細展開。

🍔 點餐系統範例

以下是點餐系統中「顧客」這個行動者的簡單 Use Case：

圖 4-1：顧客的 Use Case 圖（簡易版）

Use Case 文字描述

Use Case下單點餐
行動者顧客
前置條件顧客已登入、已選擇餐廳
主要流程1. 顧客瀏覽菜單
2. 選擇餐點並設定客製化選項（甜度、冰量等）
3. 加入購物車
4. 確認訂單內容與金額
5. 選擇付款方式並完成付款
6. 系統建立訂單並通知餐廳
替代流程3a. 購物車已有其他餐廳品項 → 提示顧客是否清空
5a. 付款失敗 → 提示重試或更換方式
後置條件訂單成立，狀態為「已下單」

📜 重點整理

  • Use Case 用橢圓表示功能，人形表示行動者
  • 每個 Use Case 代表一個獨立的使用者與系統的互動任務
  • Use Case 圖是概覽，需搭配文字描述補充細節
  • 文字描述應包含：行動者、前置條件、主要流程、替代流程、後置條件
  • 在 Chapter 5 會深入學習完整的 Use Case 圖（含多角色、Include/Extend 關係）

✎ 練習題

  • 管理菜單：新增、修改、刪除菜單品項
  • 處理訂單：接單、更新製作狀態、標記完成
  • 查看報表：查看每日/每月營收與訂單統計

其他合理答案如「設定營業時間」、「回覆顧客評價」也可以。

是的，外部付款系統（如 Line Pay API、信用卡閘道）可以是行動者。行動者不限於人，也包括與系統互動的外部系統或硬體裝置。

在 UML 中，雖然用人形符號表示，但它代表的是「與系統互動的外部實體」。

Chapter 5

5-1 環境模型與活動圖

📚 概念說明

系統塑模（System Modeling）是為系統開發各種抽象模型的程序。通常使用 UML 圖來表示。

塑模的第一步是決定系統邊界（Boundary）— 哪些功能在系統內，哪些在系統外。

活動圖（Activity Diagram）描繪業務流程中的活動順序，可用水道（Swimlane）區分不同角色的職責。

活動圖的基本元素：

  • 實心圓：活動開始
  • 圓角矩形：活動
  • 菱形：決策點
  • 箭頭：轉換線（流程方向）
  • 雙圓（牛眼）：活動結束
  • 水道：區分不同角色

📜 重點整理

  • 系統塑模的第一步是定義系統邊界，區分系統內外
  • 環境模型顯示系統與外部實體的關係
  • 活動圖描繪業務流程的活動順序
  • 水道（Swimlane）可將活動依角色分組
  • 菱形代表決策點，可以分出不同流程路徑

✎ 練習題

外部。Google Maps API 是第三方服務，不是我們開發的系統的一部分。系統會呼叫它來取得地圖和路線資訊，但它本身在系統邊界之外。

  • 實心圓：起始節點，代表流程的開始
  • 圓角矩形：活動節點，代表一個動作或工作
  • 菱形：決策節點，根據條件分支流程
  • 箭頭（轉換線）：連接活動，表示流程方向
  • 牛眼（雙圓）：結束節點，代表流程終止
  • 水道：將活動分組到不同的角色或系統

Chapter 5

5-2 Use Case 圖

📚 概念說明

Use Case 圖描繪系統與外部行動者之間所有的互動情形。它是系統功能的「全景圖」。

除了基本的行動者與使用案例之外，Use Case 之間還有兩種重要關係：

  • «include»：某個 Use Case 一定會包含另一個 Use Case（必要的子流程）
  • «extend»：某個 Use Case 在特定條件下才會擴展另一個 Use Case（選擇性的額外行為）

🍔 點餐系統完整 Use Case 圖

圖 5-3：點餐系統完整 Use Case 圖

«include» 範例：「下單點餐」一定會包含「線上付款」，因為每筆訂單都必須付款。

«extend» 範例：「下單點餐」可能被「使用優惠券」擴展，但不是每次下單都會使用優惠券。

📜 重點整理

  • Use Case 圖是系統功能的全景圖，展示所有角色與功能的關係
  • «include»：必要的子流程（虛線箭頭，從主 Use Case 指向被包含者）
  • «extend»：選擇性的擴展（虛線箭頭，從擴展者指向主 Use Case）
  • 行動者可以是人，也可以是外部系統
  • 系統邊界用虛線矩形框住所有 Use Case

✎ 練習題

是 «include» 關係。下單前必須先登入，這是必要的前置步驟。「下單點餐」include「顧客登入」。

「加購推薦」extend「下單點餐」

理由：不是每次下單都會觸發加購推薦，只有當系統判斷有合適的推薦品項時才會出現。這是選擇性行為，所以用 extend 而非 include。

5-3 序列圖

📚 概念說明

序列圖（Sequence Diagram）描述行動者與系統物件之間，以及物件彼此之間的互動順序。

序列圖有兩個軸：

  • 水平軸：參與互動的物件（由左到右排列在頂端）
  • 垂直軸：時間順序（由上到下代表時間流逝）

基本元素：

  • 生命線：物件下方的垂直虛線
  • 實線箭頭：請求訊息（呼叫）
  • 虛線箭頭：回傳訊息
  • 活化框：物件正在處理的時間段（生命線上的長方形）

🍔 點餐系統範例：下單流程序列圖

圖 5-4：下單流程序列圖

閱讀方式：由上到下看，可以清楚知道每個步驟的順序、誰呼叫誰、以及資料如何傳遞。實線代表請求，虛線代表回應。

📜 重點整理

  • 序列圖展示時間順序的物件互動
  • 水平軸排列物件，垂直軸代表時間流逝
  • 實線箭頭 = 請求訊息，虛線箭頭 = 回傳訊息
  • 活化框（長方形）代表物件正在處理中的時段
  • 適合描述單一 Use Case 的詳細互動流程

✎ 練習題

步驟 7 回傳「付款失敗」後：

  • 訂單系統將訂單狀態更新為「付款失敗」
  • 訂單系統通知顧客 App「付款失敗，請重試」
  • 顧客 App 顯示錯誤訊息給顧客
  • 步驟 10-12 不會執行（餐廳不會收到通知）

整個流程在付款系統回傳失敗後就走另一條路徑。

5-4 類別圖

📚 概念說明

類別圖（Class Diagram）是結構模型，描繪系統中的物件類別及其關係。類別包含三個區塊：

  • 類別名稱（頂部）
  • 屬性（中間）
  • 操作/方法（底部）

類別之間的四種關係

關係符號意義範例
關聯（Association）實線兩個類別互相知道對方顧客 — 訂單
繼承（Inheritance）空心三角箭頭子類別繼承父類別的屬性和方法食物、飲料 → 菜單品項
聚合（Aggregation）空心菱形整體包含部分，部分可獨立存在餐廳 ◇— 外送員
組合（Composition）實心菱形整體包含部分，部分不可獨立存在訂單 ◆— 訂單品項

多重性（Multiplicity）

標記意義
1恰好一個
0..1零或一個
*零或多個
1..*一或多個
3..5三到五個

🍔 點餐系統類別圖

圖 5-5：點餐系統類別圖

關係解讀

關係解讀多重性
顧客 — 訂單一位顧客可以有多筆訂單（關聯）1 對 *
訂單 ◆ 訂單品項訂單刪了，品項也消失（組合）1 對 1..*
訂單品項 — 菜單品項品項對應菜單上的項目（關聯）* 對 1
餐廳 ◇ 外送員餐廳關了，外送員仍存在（聚合）1 對 *
食物/飲料 → 菜單品項食物和飲料繼承菜單品項（繼承）—

📜 重點整理

  • 類別包含三區：名稱、屬性、操作
  • 關聯（實線）：兩個類別互相知道對方
  • 繼承（空心三角箭頭）：子類別擁有父類別所有屬性與方法
  • 聚合（空心菱形）：部分可獨立存在（餐廳 ◇ 外送員）
  • 組合（實心菱形）：部分不可獨立存在（訂單 ◆ 訂單品項）
  • 多重性標示兩端的數量關係（1、*、0..1、1..*）

✎ 練習題

組合（Composition）。建築物拆除了，裡面的房間也不復存在。房間無法脫離建築物獨立存在，所以用實心菱形。

父類別：付款方式（共有屬性：金額、付款時間、狀態）

  • 信用卡：卡號末四碼、有效期限、授權碼
  • Line Pay：交易編號、Line 帳號
  • 貨到付款：找零金額、收款人簽名確認

三者都繼承父類別的「金額」「付款時間」「狀態」，再各自擴充獨特屬性。

圖書館 ◇ 書本：聚合。圖書館關閉了，書本仍然存在（可以搬到別處）。

訂單和訂單編號：這不是聚合或組合關係，訂單編號是訂單的屬性（attribute），不是獨立的類別。只有當兩個獨立的類別之間有「整體-部分」的關係時，才使用聚合/組合。

5-5 狀態圖

📚 概念說明

狀態圖（State Diagram）描述系統如何回應事件，以及事件如何觸發狀態之間的轉換。

基本元素：

  • 狀態（State）：圓角矩形，系統在某個時刻的情況
  • 初始狀態：實心圓點
  • 最終狀態：牛眼（同心圓）
  • 轉換（Transition）：箭頭，格式為「事件 [條件] / 動作」

資料驅動 vs 事件驅動

比較資料驅動事件驅動
觸發方式有資料輸入時觸發處理事件發生時觸發狀態轉換
焦點資料的流動與轉換過程系統狀態的變化
常見模型資料流圖（DFD）、活動圖狀態圖
適用系統商業資料處理系統即時系統、控制系統、UI 互動
範例處理訂單報表訂單狀態流轉

🍔 點餐系統範例：訂單狀態圖

圖 5-6：訂單狀態圖

讀圖方式：每個箭頭旁邊的文字是觸發狀態轉換的事件。例如「餐廳接單」這個事件會讓訂單從「已下單」轉換到「已接單」。紅色路徑代表取消流程。

📜 重點整理

  • 狀態圖適合描述有明確生命週期的物件（如訂單、帳號）
  • 轉換格式：事件 [守衛條件] / 動作
  • 資料驅動系統關注資料如何流動，事件驅動系統關注狀態如何改變
  • 大部分商業系統是資料驅動，即時系統是事件驅動
  • 同一個系統可能同時需要兩種模型

✎ 練習題

  • 離線 → 上線（事件：打開 App 開始接單）
  • 待命中 → 已派單（事件：系統派發新訂單）
  • 已派單 → 前往取餐（事件：接受派單）
  • 已派單 → 待命中（事件：拒絕派單）
  • 前往取餐 → 等待出餐（事件：抵達餐廳）
  • 等待出餐 → 外送中（事件：取得餐點）
  • 外送中 → 待命中（事件：送達確認）
  • 任何狀態 → 離線（事件：關閉 App）

因為餐廳已經開始製作了，食材和人力已經投入。這個限制用守衛條件表示：取消事件的條件是 [尚未製作]，一旦進入「製作中」，這個條件就不成立，所以取消路徑被封鎖。

這就是為什麼在狀態圖中，取消的箭頭只從「已下單」和「已接單」出發，而不從「製作中」出發。

Chapter 5

5-6 MDA 三層模型

📚 概念說明

模型驅動式架構（Model-Driven Architecture, MDA）是一種以模型為核心的軟體設計方法。核心理念是：先建模型，再從模型自動產生程式碼。

MDA 使用三個抽象層次的模型：

  • CIM（Computation Independent Model）：計算獨立模型，描述業務領域，不涉及任何技術
  • PIM（Platform Independent Model）：平台獨立模型，描述系統功能與結構，不綁定特定技術
  • PSM（Platform Specific Model）：平台特定模型，針對特定技術平台的實作細節

🍔 點餐系統範例

圖 5-7：MDA 三層模型

具體對照

層次描述的內容點餐系統範例
CIM業務流程與領域概念「顧客從菜單選餐，付款後餐廳收到訂單開始製作」
PIM系統邏輯（類別、介面、互動）訂單類別有 計算總額() 方法，顧客與訂單是 1 對 * 關聯
PSM特定技術的實作細節用 Django ORM 定義 Order Model，PostgreSQL 儲存，REST API

MDA 的挑戰與限制

模型與實作的差距

效能優化、錯誤處理等細節很難在模型中表達，完全自動化生成程式碼仍不現實

工具生態不足

MDA 工具的易用性與整合性有限，且公司不想依賴可能退出市場的小型工具商

敏捷方法的競爭

敏捷開發強調快速迭代與直接寫程式碼，與 MDA 先建模型的理念形成對比

適用範圍有限

MDA 較適合大型、長壽命系統，小型或快速變化的專案效益不明顯

現實情況：MDA 一直沒有成為軟體開發的主流方法。對大多數系統而言，需求工程、安全性、與舊系統整合、以及測試，比自動產生程式碼來得更重要。

📜 重點整理

  • MDA 以模型為核心，目標是從模型自動生成程式碼
  • 三層模型：CIM（業務）→ PIM（邏輯）→ PSM（實作）
  • PIM 的優勢是平台獨立，同一個模型可轉換到不同技術平台
  • 實務上完全自動化轉譯仍不可行，MDA 未成為主流
  • MDA 較適合大型、長壽命系統，不適合敏捷快速迭代的專案

✎ 練習題

屬於 PIM。它描述的是系統的效能需求（非功能需求），但沒有指定用什麼技術實現。如果改成「使用 Redis 快取加速訂單建立，回應時間 ≤ 2 秒」，那就是 PSM。

  • 模型到程式碼的差距：許多細節（效能優化、例外處理）無法在模型中充分表達，完全自動生成不實際
  • 敏捷方法興起：業界轉向快速迭代的敏捷開發，直接寫程式碼比先建完整模型更有效率
  • 工具不成熟：MDA 工具不夠好用，公司不想依賴可能消失的小廠商
  • 實作不是主要問題：軟體開發的困難在於需求、安全性、舊系統整合、測試，而非程式碼撰寫本身
